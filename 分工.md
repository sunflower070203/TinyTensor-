### A — 基础版原理讲解

**【张量与计算图 0:00-1:00】**

> 我来讲解基础版的实现原理。
>
> 我们的 Tensor 类内部持有一个共享指针指向 Node 结构体。Node 保存四个信息：数据、梯度、父节点列表、以及一个反向传播的 lambda 闭包。
>
> 每次运算，比如 a + b，会创建一个新的 Node，把 a 和 b 记录为父节点，同时注册一个闭包。这个闭包的作用是：当调用 backward 时，把输出的梯度累加到 a 和 b 的梯度上。

**【反向传播 1:00-2:00】**

> 调用 loss.backward() 时，系统先从 loss 节点做拓扑排序，确保父节点排在子节点前面。然后逆序执行每个节点的闭包，梯度就从 loss 逐层传回到输入。
>
> 这里有一个关键细节：梯度用 += 累加而不是 = 赋值。因为在计算图中一个节点可能被多个运算使用，梯度需要求和。这和 PyTorch 的行为一致。
>
> 另一个细节是广播。比如标量 bias 加到一个 batch 的输出上，bias 的梯度等于整个 batch 对应梯度之和。我们的 add_broadcast_grad 函数处理了这种情况。

### B — 挑战版可视化讲解

**【可视化原理 0:00-1:00】**

> 我来讲解挑战版的可视化实现。
>
> 决策边界的生成原理是：在二维平面上取一个 200 乘 200 的网格，对每个点做一次前向预测，得到一个 0 到 1 的概率值。然后把概率映射为颜色：0 是蓝色，0.5 是白色，1 是红色。这样就得到了一张热力图。
>
> 图片格式我们选择了 BMP，因为 Windows 双击就能打开，不需要任何额外软件。整个图片生成是纯 C++ 实现的，直接写入二进制文件，零依赖。

**【实时窗口 1:00-2:00】**

> 实时训练窗口使用 Windows 原生的 GDI API。我们创建一个 800 乘 600 的窗口和一个像素缓冲区，训练过程中每 50 个 epoch 重新计算一次决策边界并刷新窗口。
>
> 为了让窗口不卡死，我们在训练循环中每隔 5 个 epoch 调用一次 yield 函数，让出 CPU 时间并处理窗口消息。
>
> 现在请看演示——（运行 train_live.exe）——可以看到决策边界从一开始的随机状态，随着训练逐渐变成清晰的分类面。这个过程是实时的，窗口可以拖动、可以关闭。

## 预备 Q&A（老师可能问的问题）

| 问题 | 参考回答（由孙浩主答，A/B 补充） |
|------|--------------------------------|
| 为什么不用 Python？ | 题目要求 C++ 实现，而且纯 C++ 零依赖更能体现对底层原理的理解。 |
| 梯度计算怎么验证正确性？ | tests.cpp 中的 test_scalar_autograd 用解析梯度对比验证；另外可视化本身也能直观验证——如果梯度算错，决策边界不会收敛。 |
| 为什么用 MSE 而不是交叉熵？ | MSE 实现简单，对二分类任务已经足够收敛。如果需要可以扩展，但会增加 tensor.hpp 的复杂度。 |
| 广播是怎么实现的？ | 前向时用 value_at 按维度取值，反向时用 add_broadcast_grad 把梯度沿广播维度折叠求和。 |
| 实时窗口为什么不会卡？ | 每 5 个训练步调用 Sleep(0) 让出 CPU，每 50 个 epoch 刷新一次决策边界，决策边界降到 200×200 分辨率。 |
| 计算图会不会内存泄漏？ | 不会，所有节点用 shared_ptr 管理，闭包也捕获 shared_ptr，只要输出 tensor 存活，子图就不会被释放。 |
